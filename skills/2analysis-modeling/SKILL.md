# 赛题分析与建模设计

## 数学建模规范参考

如需领域判断，读取 `../_references/math_modeling_norms.md` 中的“赛题理解与子问题识别”“假设与模型建立”和“题型防错速查”小节。该文件只作为规范知识库，不替代本阶段的产出结构。
如需确认跨阶段交接字段、统一状态语义或回退重置规则，读取 `../_references/workflow_contract.md`；本阶段只维护分析产物和建模交接。

## 必须产出

在当前工作目录的 `reports/` 下创建或更新：

- `reports/ANALYSIS_MODELING_REPORT.md`
- `reports/contracts/model_tasks.json`

默认统一使用 `reports/ANALYSIS_MODELING_REPORT.md` 作为正式产物；旧项目若仍保留 `PROBLEM_ANALYSIS.md`，只作为兼容参考，不再作为默认输出名。

## 工作流程

### Step 1: 子问题拆解

只把题面中明确编号的顶层问题当作子问题，例如“问题一/二/三”或 “Problem 1/2/3”。不要把小问、背景描述、数据说明、提交要求误当成独立子问题。

在 `reports/ANALYSIS_MODELING_REPORT.md` 开头明确写出：

- 本赛题共有多少个子问题。
- 每个子问题的输入、输出、目标、约束、依赖关系。
- 计划展示哪些图表或表格。

### Step 2: 假设敏感性预检

列出关键歧义，不要急着定模型。对影响结果的歧义至少给出两种解释，并用简单验算或逻辑递进判断采用哪一种。

报告中至少包含：

```markdown
## 假设敏感性预检

### 模糊表述及解释
### 快速验算与递进性检查
### 最终采用的解释
### 对后续结果和图表的影响
```

### Step 3: 数据理解与总体路线

对每份附件说明：

- 行列规模和字段解释。
- 缺失、异常、重复、单位不一致。
- 可直接用于建模的变量。
- 需要派生的指标。

然后给出从题面到结果的总体路线，确保代码阶段能直接承接。

### Step 4: 写建模报告

在 `reports/ANALYSIS_MODELING_REPORT.md` 中写出可交给代码阶段实现的完整方案。每个子问题至少包含：

- 问题目标
- 符号和变量
- 模型假设
- 目标函数
- 约束条件
- 求解方法
- 输入输出
- 代码实现要点
- 结果校验方法

如果子问题数量不是 3 个，按实际题面调整章节，不要硬凑。

### Step 5: 写机器可读交接契约

在 `reports/contracts/model_tasks.json` 中记录最小交接信息，优先保持扁平、可读、可手改：

```json
{
  "subproblems": [
    {
      "id": "q1",
      "title": "子问题标题",
      "goal": "要求解什么",
      "inputs": ["数据源或字段"],
      "model": "采用的模型/算法",
      "outputs": ["结果文件或表格"],
      "key_metrics": ["误差/目标值/约束指标"],
      "downstream_tasks": ["代码阶段需要完成的动作"]
    }
  ]
}
```

不要把它写成复杂 schema；只记录代码阶段真正需要的任务边界。

## 质量要求

- 所有结论都能回到题面或数据。
- 不编造数据字段和数值。
- 不跳过歧义分析。
- 模型既要有数学表达，也要能被代码实现。
- 若数据不足或题面不清，要明确记录风险和替代方案。
